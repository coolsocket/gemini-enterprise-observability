# Findings — GE 生成能力 API 探测 (2026-07-03)

## TL;DR

| 能力 | SA 直接调 API 可以吗 | 备注 |
|---|---|---|
| **Chat (StreamAssist)** | ✅ **完全可以** | 返回流式文本响应, 音审计日志正常写入 |
| **图片生成 via chat prompt** | 🟡 **能触发, 下载 API 存在但被 workforce-identity gate 挡** | GE 后端用 Nano Banana 2 生成图, 响应给 `fileId + mimeType`; 下载端点 (`:listFiles`, `:getFile`, `:downloadFile` 等 colon-verb) 都存在, 但均 403 "Session is not owned by the provided user" —— 跟 NotebookLM 同一个 workforce gate |
| **视频生成 via chat prompt** | 🟡 同上 | 同上, 生成成功但下载被 gate 挡 |
| **Deep Research (AsyncAssist)** | ❌ **404 无公开 API** | v1alpha 里根本没这个方法, 仅 UI 走内部 v1main 调用 |
| **直接调 Vertex Imagen** | ❌ **403 (可修)** | SA 缺 `aiplatform.endpoints.predict`, 授权后可用 |

## 详细结果

### 1. Chat — ✅ works perfectly

```bash
POST /v1alpha/.../assistants/default_assistant:streamAssist
Body: {"query": {"text": "What is 2+2?"}, "session": "-"}
```

Response (streaming JSON chunks):
```json
{"answer": {"state": "SUCCEEDED", "replies": [
  {"groundedContent": {"content": {"text": "4"}}}
]}}
```

- ✅ Session ID auto-generated (`session: "-"` triggers auto-create)
- ✅ Multiple chunks stream in (thoughts, answer, suggestions)
- ✅ Audit log records `AssistantService.StreamAssist` under principal SA

### 2. Image via chat prompt — 🟡 生成 OK, 下载 API 存在但被 workforce-identity gate 挡

Prompt: *"Generate an image of a frog wearing a top hat, cartoon style"*

Response (last content chunk before SUCCEEDED):
```json
{"role": "model", "file": {
  "mimeType": "image/png",
  "fileId": "GENERATED_FILE_UUID"
}},
{"role": "model", "text": "Image generated by Nano Banana 2."}
```

**GE 后端确实生成了图** (Nano Banana 2 model), 响应给出 `fileId + mimeType + 生成模型 + assistToken + answer.name`。Session 自动命名为 "Frog with top hat cartoon"。

#### 找下载 API 的完整过程 (重要 —— 之前判断错了)

Discovery doc 里的 `sessions.files.list` 是**subresource 语法** (`.../sessions/{sid}/files`), 请求这个 URL 一律返回 `404 "cannot be resolved"`。之前据此以为下载 API 不存在。

但**换成 colon-verb 语法**就完全不同了:

| URL | HTTP | 结论 |
|---|---|---|
| `GET .../sessions/{sid}/files` (doc 说的路径) | 404 "cannot be resolved" | 路由不认 |
| `GET .../sessions/{sid}:listFiles` | **403 "Session is not owned by the provided user"** | **endpoint 存在** |
| `GET .../sessions/{sid}:getFile` | 403 同上 | 存在 |
| `GET .../sessions/{sid}:downloadFile` | 500 "Internal error" | 存在, 需要 body 参数 |
| `GET .../sessions/{sid}:fetchFile` | 403 同上 | 存在 |
| `GET .../sessions/{sid}:media` | 403 同上 | 存在 |
| `GET .../sessions/{sid}:getContent` | 403 同上 | 存在 |
| `GET .../sessions/{sid}:listContent` | 403 同上 | 存在 |
| `GET .../sessions/{sid}:getMedia` | 403 同上 | 存在 |
| `GET .../assistAnswers/{aid}` | 404 "Method not found" | 不存在 |

**所以 download API 是有的**, 只是 UI 内部走的路径 (colon-verb) 和 discovery doc 里 declarative 的路径 (subresource) 不同。

#### 挡住的门是 workforce identity, 不是 IAM

再深入: 试图跨过 `Session is not owned by the provided user`, 试了以下都无效:
- ❌ 用 `sessions.patch` 把 `userPseudoId` 改成 SA 的 email
- ❌ 用 `sessions.create` 显式创建带 `userPseudoId` 的 session, 再在同一 session 里 streamAssist + list
- ❌ 用 `sessions.create` 创 session, 传数字型 pseudoId (类似 UI 那种)
- ❌ 把 `x-goog-assist-token` header 传上去 (assistToken 是每次响应给的一个 token, 想过是不是"证明我是发起者")
- ❌ 用真人 OAuth token 而非 SA (**真人也 403** —— 我的 personal user token 同样被拒)

**结论**: "Session owner" 的 identity check 用的是**workforce identity**, 跟 IAM / userPseudoId / OAuth 都无关 —— 完全一致于 NotebookLM 的 gate ([参考 notebooklm-sa-gate.md](../de-api-probe/notebooklm-sa-gate.md))。

#### `FileMetadata` schema 佐证下载 API 就是这个

Discovery doc 里 `GoogleCloudDiscoveryengineV1alphaFileMetadata` 的字段:
```
{views, uploadTime, lastAddTime, metadata, fileId, name, fileOriginType,
 originalSourceType, downloadUri, mimeType, originalUri, byteSize}
```

有 `downloadUri` 字段 —— 说明这个 API **原本设计**就是要返回签名 URL 让你下载的。只是这个字段只在 owner 身份通过后才能拿到。

→ 修正结论: **下载 API 存在, GE 也真的生成了文件, 但拿这个 API 需要 workforce identity (跟 UI session 挂钩), SA/OAuth user token 都过不了 ownership 检查**。跟 NotebookLM 是同一个门。要程序化拿到文件, 目前**没有已知路径**。

### 4. Deep Research (AsyncAssist) — ❌ 404

```
POST /v1alpha/.../assistants/default_assistant:asyncAssist
→ 404 "Method not found"
```

v1alpha discovery doc 里 grep 所有 assist 方法, 只找到 `streamAssist`。**AsyncAssist 不是公开 API**, 只有 UI 通过 v1main 内部端点调用。审计日志里能看到 AsyncAssist 事件是因为 UI 触发了它, 但**你没法从代码触发**。

> 🤔 有意思: 探针里 AsyncAssist 返回 404, 但 audit log 里**还是**出现了一条 `AssistantService.AsyncAssist` 记录 (23:24:35), 说明 404 是路由层拒绝, 但服务端仍记录了尝试。

### 5. Vertex Imagen 直连 — ❌ 403 (可修)

```
POST https://us-central1-aiplatform.googleapis.com/v1/.../imagegeneration:predict
→ 403 "Permission 'aiplatform.endpoints.predict' denied"
```

不是产品限制, 只是 SA 没授 Vertex predict role。授了就能用 — 但**这跟 GE 是两码事**, 用了不走 GE quota, 也不出现在 GE audit log 里。

## 对我们 dashboard 的启示

### 我可以更新的观察: image_gen / video_gen 精度

之前 README 说 "GE 图/视频生成后端跑在 Google infra, 不发 customer audit log"。**这个说法半对半错**:

- ✅ **对**: 生成过程本身 (Nano Banana 2 的实际调用) 不出现 audit log
- ❌ **错**: 触发它的 StreamAssist 请求**是**有 audit log 的, 只是 audit log 里看不出这次 StreamAssist 到底生成了啥

所以我们的 prompt 关键词启发不是唯一路径 —— **如果我们能拿到 StreamAssist 请求的完整 payload** (data_write audit + payload capture), 就能看到 prompt 里"generate an image" 这种意图, 也能间接看到响应里的 fileId。但当前 audit log 只记 method name, 不记 payload, 所以关键词启发仍是最好办法。

### 我可以更新的观察: session 自动命名

我们之前没利用 GE 给 session 自动生成的 displayName ("Frog with top hat cartoon", "Ocean waves at sunset video")。这些**非常清晰**地反映了每个 session 的意图。用 `ListSessions` API (`.../engines/{engine}/sessions`) 可以拉到, 加到 dashboard 上会是很好的 session-level summary。

**Dashboard TODO**: 在 Conversations 页里, 把 sessionId → sessionDisplayName mapping 加进来, 单条对话就有语义化标题了。

## 对"能不能给别人 API 集成"的最终结论

假设别人的问题是:  *"我能不能在我自己的应用里嵌 GE, 让用户对话/生图/生视频, 全走 GE API?"*

回答:
- ✅ Chat: 可以, 完全能嵌
- ⚠ 图/视频: **能触发生成, 但拿不到生成的文件**, 无法真的展示给终端用户。要么用 GE 的 embedded UI widget, 要么退回 Vertex 直连 (自己管 quota, 不进 GE 报表)
- ❌ Deep Research: 完全不能, 只有 UI
